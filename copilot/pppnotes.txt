PPP Presentation Speaker Notes
Slide 1: Title Slide
What to say:

"I'm investigating GPU-aware architecture patterns - specifically what makes games work well with modern upscaling tech like DLSS and FSR"
Keep it brief, this is just the intro
If asked about "temporal upscaling":

Temporal upscaling uses data from previous frames to reconstruct higher-resolution images
DLSS uses AI/neural networks, FSR uses hand-crafted algorithms
Both need consistent frame-to-frame data or they break
Slide 3: Problem Domain
Key points to emphasize:

This isn't theoretical - student projects actually break upscalers all the time
The problem is lack of documented patterns, not lack of hardware capability
AAA studios know this stuff but it's not in textbooks or academic papers
If asked "Why does this happen?":

Upscalers assume your code produces stable, predictable output each frame
Frame-rate-coupled logic means simulation changes at different FPS
Multi-writer systems create race conditions that vary frame-to-frame
These create temporal discontinuities that upscalers can't handle
If asked about specific artifacts:

Ghosting = trailing images from previous frames not integrating correctly
Smearing = motion data is inconsistent, upscaler interpolates incorrectly
Stuttering = frame times vary wildly, breaking temporal assumptions
Slide 4: Why It Matters
Key message:

There's a knowledge gap between industry and education
Hardware vendors (Nvidia, AMD) document their APIs but assume you already know the software patterns
This project bridges that gap with empirical validation
If challenged on "Is this really a problem?":

Point to any student game project using UE5 or Unity with DLSS enabled
Most have visible artifacts because they don't know these patterns
Even shipped indie games struggle with this
If asked "Why can't students just read Nvidia docs?":

Nvidia docs explain HOW to integrate DLSS, not how to structure your entire codebase
The patterns (fixed-timestep, etc.) exist but aren't connected to "this is required for upscalers"
Slide 5: The Solution
Emphasize the two-part structure:

Research paper establishes the "why" - how hardware changes affect software
Code project demonstrates the "how" - implementing patterns in practice
They reinforce each other: research informs implementation, implementation validates research
If asked "Why not just do code?":

Without research foundation, it's just "here's some patterns"
Research provides context on WHY these patterns emerged (hardware constraints)
Makes it defensible as an academic capstone, not just a coding exercise
If asked about the three patterns:

Fixed-timestep: Separates physics/game logic (60 or 120 Hz) from rendering (variable)
Workload budgeting: AI gets X milliseconds per frame, no more, prevents spikes
Temporal coherence: One system owns motion data, prevents conflicting updates
Slide 6: Research Foundation
Key scholarly angle:

Not asking "are upscalers good?" - asking "how does hardware evolution reshape software engineering?"
Focuses on Blackwell (RTX 50 series) because it doubles down on AI integration
This is architectural analysis, not performance benchmarking
If asked about methodology:

Literature review of GPU architecture papers (Nvidia technical docs, academic papers on rendering)
Case study analysis of how UE5 and Unity adapted their architectures
Pattern extraction - identifying what changed in development workflows
If asked "What's novel here?":

Not claiming to invent patterns - they exist in AAA
Novel contribution is CONNECTING hardware evolution to pattern adoption
Plus empirical validation that students can actually use
Slide 7: Research Areas
GPU Evolution:

Blackwell has dedicated tensor cores, ray tracing cores, AND traditional CUDA cores
Ada Lovelace (RTX 40) started this, Blackwell expands it
Hardware assumption shift: rendering is now AI-assisted, not purely rasterized
Case Studies:

UE5 Lumen = software ray tracing that works without RT cores (but benefits from them)
UE5 Nanite = virtualized geometry, needs stable frame pacing
Unity DOTS = data-oriented design for parallel processing
All emerged in response to modern GPU capabilities
Pattern Analysis:

What did developers do BEFORE these GPUs? (frame-coupled logic was fine at 60 FPS)
What changed? (variable refresh rates, AI upscaling requirements)
How do pros handle it vs. students? (pros have established codebases, students build from scratch)
Slides 8-13: Features 1-6
For each feature, be ready to explain:

Feature 1 (Fixed-Timestep):

Game logic runs at fixed intervals (e.g., every 16.67ms for 60 Hz simulation)
Rendering runs as fast as possible (could be 120 FPS, 144 FPS, variable)
Interpolation smooths visual state between simulation ticks
Why it matters: DLSS expects stable motion vectors, not ones that change based on frame rate
Feature 2 (Workload Budgeting):

AI gets a time budget (e.g., 3ms per frame)
If pathfinding takes longer, it's time-sliced across multiple frames
Prevents "AI thinks hard = frame drops to 15 FPS"
Why it matters: Frame-time variance breaks temporal reconstruction
Feature 3 (Temporal Coherence):

One system writes motion data, no conflicts
Effects use delta-time, not frame counts ("fade over 2 seconds" not "fade over 120 frames")
Why it matters: Upscalers need frame N to be predictable based on frame N-1
Feature 4 (Performance Measurement):

Not just average FPS - need 1% lows, 0.1% lows (worst-case performance)
Frame-time variance is critical metric
Memory tracking because allocation spikes cause frame hitches
Why it matters: Can't validate patterns without measurements
Feature 5 (DLSS/FSR):

Integrating the actual SDK, not just theory
Testing patterns with upscaling ENABLED to see if they actually work
Why it matters: Proves the patterns help in practice
Feature 6 (Game Prototype):

Need realistic stress-test environment
AI provides variable workload
Environmental effects test temporal stability
Why it matters: Academic exercise without realistic context isn't useful

Slide 17: Technical Architecture
Be ready to explain the diagram:

Fixed-timestep game loop runs simulation
Player and AI systems feed into workload budgeter
Budgeter ensures AI doesn't blow the frame-time budget
Motion system has single-writer authority (temporal coherence)
Everything connects to show how patterns interact
If asked "Why this architecture?":

Separates concerns: simulation vs. rendering
Prevents common pitfall of tightly-coupled systems
Matches how AAA engines structure their game loops
Data flows one direction, no circular dependencies
Slide 18: Validation Methodology
Key change from earlier plan:

NOT building baseline + optimized versions
Applying research findings directly
Measuring whether patterns meet target metrics
If asked "How do you know the patterns work?":

Measuring frame-time variance (should be <5ms std dev)
Checking motion vector stability (>95% coherent)
Visual inspection with DLSS/FSR enabled (minimal artifacts)
Comparing against established metrics from shipped games
Slide 19: Testing Strategy
Performance Testing:

Frame-time consistency = all frames take similar time to render
1% lows = worst 1% of frames, shows edge cases
CPU/GPU utilization = making sure not bottlenecked
Visual Quality:

DLSS/FSR artifacts are subjective but recognizable
Motion vectors can be visualized in-engine
Screenshot comparisons show before/after
Stress Testing:

Variable AI workload = sometimes pathfinding, sometimes idle
Environmental effects = particles, lighting changes
Peak load = everything happening at once
Slide 20: Success Metrics
Frame-time variance <5ms:

Consistent frame pacing is critical for upscalers
<5ms std dev means frames are predictable
Motion vector stability >95%:

Most frames have coherent motion data
Upscalers can interpolate correctly
DLSS/FSR quality:

Minimal ghosting means temporal data is stable
"Minimal" is qualitative but recognizable
1% low FPS within 10% of average:

Worst-case performance isn't dramatically worse
Indicates stable workload management
If challenged on these metrics:

These are based on shipped game standards
Not arbitrary - they reflect industry expectations
Research will cite sources for these targets

Engines provide tools, not architectural guidance

This project shows HOW to use those tools correctly
"Has this been done before?"

AAA studios do this, but don't publish their methods
Academic papers on upscaling focus on algorithms, not software engineering
This fills the gap: practical patterns for developers
"What if your patterns don't work?"

Research paper documents WHY they should work (theoretical foundation)
If implementation fails, that's still a valid finding
Can analyze what went wrong and why
"Why these three patterns specifically?"

They're the most commonly cited in professional talks (GDC, Unreal Fest)
They address the three main upscaler requirements (stable timing, predictable workload, coherent motion)
They're implementable in a 6-week timeframe
"What GPU are you testing on?"

Consumer-grade GPU (RTX 30 or 40 series most likely)
