KNOWN FLAWS BY DEMO — MUST FIX BEFORE FINAL SUBMISSION
────────────────────────────────────────────────────────────────────────────────

── DEMO 1 — Motion Authority (MoverActor / MotionAuthorityActor) ─────────────

  FLAW 1: Logger Init() called once per actor, not once per session.
    Each MoverActor calls FMotionLogger::Get().Init() in its own BeginPlay.
    Init() is guarded by bInitialized so only the FIRST actor to fire wins.
    If a direct-mode actor fires before an authority actor, the session is
    correctly started, but the order is non-deterministic across Play sessions.
    FIX: Move Init() call to a GameMode or GameInstance BeginPlay so it fires
    exactly once before any actor Tick. Remove Init() calls from all actors.

  FLAW 2: Flush() is only called from MotionAuthorityActor::Tick.
    If the level has ONLY direct-mode MoverActors (no MotionAuthorityActor),
    the buffer is never flushed to disk — the CSV will be empty or truncated.
    FIX: Also call FMotionLogger::Get().Flush() from MoverActor::EndPlay or
    hook it into a game-wide EndPlay delegate.

  FLAW 3: Authority-mode mover logs nothing directly.
    When bUseAuthority == true, the mover silently submits to the manager and
    the manager logs. But if the manager has bAuthorityMode toggled OFF mid-
    session, submitted inputs are discarded and nothing is logged for that frame.
    FIX: Log a "dropped" row with mode=Dropped in MotionAuthorityActor when
    bAuthorityMode is false, so the CSV fully reflects every frame.

  FLAW 4: MoverActor uses its own TimeElapsed for oscillation phase.
    Two actors placed at the same time will be in sync, which looks good.
    But if actors are placed at different world origins or BeginPlay fires
    at different frames, they start out of phase arbitrarily.
    FIX: Use GetWorld()->GetTimeSeconds() instead of a local TimeElapsed
    accumulator (same fix already applied in MotionModelActor).

  FLAW 5: BasicShapeMaterial may not have a "Color" vector parameter.
    DynMaterial->SetVectorParameterValue(TEXT("Color"), ...) silently does
    nothing if the parameter name doesn't exist. Green/red coloring may not
    work at all without a material that exposes a Color parameter.
    FIX: Create a simple M_DemoBase material in the Content folder with a
    Color vector parameter wired to Base Color. Reference it instead of
    /Engine/BasicShapes/BasicShapeMaterial.

── DEMO 2 — Fixed Timestep (TimestepActor) ──────────────────────────────────

  FLAW 1: Spike injection replaces DeltaTime rather than adding to it.
    EffectiveDt = SpikeDeltaTime completely discards the real frame time.
    This means the spike fires on a frame that might also be a normal 16ms
    frame — the total simulated time is wrong.
    FIX: EffectiveDt = DeltaTime + SpikeDeltaTime to simulate a hitch on top
    of a real frame, which is what an actual game freeze looks like.

  FLAW 2: Both actors share bSimulateSpikes independently.
    Each TimestepActor has its own SpikeTimer. If they are placed with the
    same SpikeInterval, their spikes fire at the same time, meaning the fixed
    actor also gets its accumulator flooded simultaneously. The contrast is
    still visible but the fixed actor's many sub-steps may cause a visible
    position jump too (not an error, but confusing to an audience).
    FIX: Offset the variable actor's SpikeInterval slightly (e.g. 5.0 vs 5.3)
    or add a SpikePhaseOffset property. Alternatively, drive spikes from a
    shared manager so they fire identically for both actors simultaneously
    (making the comparison fair and dramatic).

  FLAW 3: StepSpring comment still says "Damped spring" after damping was
    removed from defaults.
    FIX: Update the comment to "Undamped spring (symplectic Euler)" to be
    accurate and to correctly describe the integrator being used.

  FLAW 4: Fixed timestep uses forward Euler per sub-step, not symplectic Euler.
    Forward Euler applied in sub-steps is better than one large step, but it
    is still slightly energy-gaining. For a truly stable fixed-timestep
    integrator the velocity should be updated first:
      Velocity += Accel * Dt;
      Displacement += Velocity * Dt;   ← uses NEW velocity (symplectic)
    The current code happens to do this already (velocity updated before
    displacement). This is correct — but it should be explicitly documented
    as symplectic Euler to be academically defensible.

── DEMO 3 — Time-based vs Frame-based Motion (MotionModelActor) ──────────────

  FLAW 1: SpikeDeltaTime is injected as EffectiveDt but the time-based actor
    completely ignores EffectiveDt — it uses GetWorld()->GetTimeSeconds().
    This means spikes have ZERO visual effect on the time-based actor, which
    is actually correct behaviour, but the EffectiveDt variable is computed
    and stored for the time-based path and then never used. This is dead code
    that will confuse future developers.
    FIX: Only compute/apply EffectiveDt for the frame-based path. For the
    time-based path, remove EffectiveDt entirely and just read WorldTime.

  FLAW 2: Frame-based mode ignores AssumedFPS when running above that FPS.
    If the actual FPS is 120 and AssumedFPS is 60, the frame-based actor
    oscillates at DOUBLE speed (phase advances twice per oscillation period).
    This makes the "bad" actor look wrong in both directions (too slow at low
    FPS, too fast at high FPS) which is correct, but the demo should make
    this explicit in the on-screen text.
    FIX: Add actual FPS to the on-screen debug message:
    "[FrameBased] <name> | x=X | actual=YHz assumed=60Hz"

  FLAW 3: Both actors start at the same phase (0) regardless of their
    AssumedFPS, so they start in sync and diverge only over time.
    The divergence is the demo's whole point but it takes several seconds to
    become visible. FIX: Start the frame-based actor slightly out of phase
    (e.g. FramePhase = PI/4) so the visual drift is immediate.

── DEMO 4 — Workload Budgeting (WorkloadActor) ───────────────────────────────

  FLAW 1: The unbudgeted actor may not cause a visible frame hitch on a fast
    machine. TasksPerFrame=80 * IterationsPerTask=5000 = 400k FMath::Sin/Cos
    calls. On a modern CPU this may run in <1ms total, producing no observable
    DeltaTime spike.
    FIX: Raise IterationsPerTask default to 50000 or add a bForceHeavyLoad
    bool that bumps iterations to 200000 for presentation mode. Alternatively
    add an artificial sleep (FPlatformProcess::Sleep) to simulate the hitch
    without CPU math.

  FLAW 2: The budgeted actor's queue grows unboundedly if budget is too small.
    Each frame adds TasksPerFrame tasks. If the budget allows processing fewer
    than TasksPerFrame per frame, the deferred count grows without limit.
    There is nothing preventing the TArray from growing to millions of entries
    over a long play session.
    FIX: Add a MaxQueueSize cap. If the queue exceeds it, drop the oldest
    entries and log a warning: "WorkloadActor: queue overflow, dropping N tasks."

  FLAW 3: ProcessTask() uses a volatile loop but the compiler may still
    partially optimize it in Development config. Shipping config will
    definitely optimize it away.
    FIX: Use FPlatformProcess::Sleep(0.0001) as an alternative or keep the
    volatile but add a comment explaining why, and note this may behave
    differently across build configurations.

  FLAW 4: FrameWorkMs is measured from the start of the task loop, but the
    position update and SetActorLocation happen AFTER the timer check.
    The logged FrameDeltaCm and the FrameWorkMs are out of sync by one frame.
    FIX: Record PrevPos before the task loop and measure NewPos after
    SetActorLocation but before the log call — same order as is done in
    TimestepActor.

── FMOTIONLOGGER — Cross-cutting ─────────────────────────────────────────────

  FLAW 1: Singleton is never reset between Play sessions in the editor.
    bInitialized stays true after PIE ends. The next Play session calls
    Init() which returns early — so all subsequent sessions log to the SAME
    file as the first session (or not at all if the file handle state is stale).
    FIX: Add a Reset() method that sets bInitialized = false and clears
    FilePath/Buffer. Call it from a UGameInstance::Shutdown override or
    from a static delegate on FWorldDelegates::OnWorldBeginTearDown.

  FLAW 2: ActorName may contain commas if the user renames actors in the
    editor (e.g. "Mover, Left"). This would corrupt the CSV.
    FIX: Quote the ActorName field or replace commas:
    ActorName.Replace(TEXT(","), TEXT(";"))

  FLAW 3: The CSV has no column for DemoName/LevelName. If data from multiple
    levels is analyzed together, there is no way to filter by demo.
    FIX: Add a LevelName column populated from GetWorld()->GetMapName().
